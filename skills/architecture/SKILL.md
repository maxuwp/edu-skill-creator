# PAGE Stage 3: Architecture

Turns the approved intent + grounding map into a buildable design. The step sequence
below applies the official plugin-dev plugin's creation phases 2–3 (component planning →
detailed design) — steps 1–4 are its component planning, steps 5–9 its detailed design —
hardened by L3–L6, L8, L9. Outputs: `architecture.md` and the new plugin's
`BUILD_PLAN.md` (resumable checklist), both gate-approved.

**Inputs (exact):** `intent.md` with an approved `intent_gate_decision.json`, and the
draft `grounding_frameworks.md` with an approved `grounding_gate_decision.json`, from
the build session. **Refuses to run** if either file is missing, unapproved, or marked
stale — halt and name the unresolved stage instead.

## Design the pipeline

1. **Stages.** Finalize the stage list from the grounding map (every stage keeps its
   anchor). For each stage: purpose, inputs (exact files/manifest fields), outputs
   (exact files), refusal conditions (missing or stale inputs).
2. **Decompose fat gates (L4).** Any stage whose gate would carry more than one decision
   becomes a wizard of narrow steps (the POSED Stage-3 six-step wizard is the worked
   example). Check the total gate load against the intent's gate budget; if over, design
   a staged (default) vs. express mode rather than silently dropping gates.
3. **Drafter/reviewer pairings (L3).** Every content-generating step gets an independent
   reviewer: fresh subagent session, input allowlist, rubric (/100, threshold 85,
   critical flags), durable review log written before the human page. Name each rubric
   file now; `page-draft` authors them.
4. **Gates (L5).** Specify every gate with the full gate-spec table from
   `<page-skill-dir>/reference/gate_design_patterns.md` (gate_id, decision, artifact +
   stable-id scheme, reviewer, decision_file schema, owns, invalidates, consent).
5. **Dependency model (L4).** Define the manifest: state machine per artifact,
   `valid_from_*`, `stale_due_to`, `needs_regeneration`, `superseded_by`; which
   approvals invalidate what; compile/assembly blocked while stale; stable item-id
   scheme per artifact type.
6. **Contested choices at intake (L2).** Place every inventoried choice as an explicit
   intake question with its neutral introductions; thread the chosen stance through the
   stages it affects; add a reviewer critical flag "stance violated" where relevant.
7. **Cost ladder (L6).** Mark token-expensive stages (simulations, sweeps) with their
   full/lite/skip consent design and the manifest field recording the mode.
8. **Data-flow & security model (mandatory when applicable).** If the plugin handles
   student work, grades, or identifiable student information, calls external APIs or
   vendors, or generates HITL web pages, the architecture MUST include a data-flow
   model: what data enters each stage, where it rests (files, logs, decision JSON),
   what leaves the machine (to which service, disclosed where, under what consent),
   retention/deletion points, redaction rules for logs, and who may run/read what —
   grounded in the intent's Part A.7–A.8 profile and the grounding library's
   privacy/security/accessibility section (FERPA/PTAC, NIST SSDF, WCAG 2.2). Generated
   HITL pages are UI: specify their keyboard and screen-reader operability here, not as
   an afterthought.
9. **Component inventory.** Beyond skills: agents, gate app (adapt POSED's guided app —
   see gate_design_patterns.md "Session mechanics"), scripts, reference files. Decide
   what is generated by `page-scaffold` vs. authored by `page-draft`.
10. **Lifecycle stages (L9).** Include reflect and refresh stages in the design by
   default; the author may decline them at the gate (their call — L2 applies to us too).

## Output — `architecture.md`

Sections: Stage map (table: stage → skill name → anchor → inputs → outputs → gate) ·
Gate specs · Dependency model & manifest schema · Reviewer pairings & rubric list ·
Contested-choice threading · Data-flow & security model (or an explicit "not
applicable: touches no student data, no external services, no generated UI") ·
Component inventory · Deferred ideas.

## Output — the new plugin's `BUILD_PLAN.md` (L8)

A checkbox list covering every file the build will create, in dependency order
(scaffold → references → rubrics → skills → scripts → docs → lint → link → commit), with
the rule stated at the top: *mark items as they land; resume at the first unchecked
item.* PAGE's own `docs/BUILD_PLAN.md` is the format example.

## Independent review (before the author sees the design — L3)

`architecture.md` is a drafted artifact, so it gets the same treatment this skill
mandates for every stage it designs. Dispatch a fresh subagent session whose inputs are
ONLY: the draft `architecture.md`, the approved `intent.md`, the approved grounding map,
and `gate_design_patterns.md`. It checks: every gate spec complete (all eight fields),
dependency model coherent (no artifact without an owner or invalidation path), every
contested choice threaded from intake to the stages it affects, gate load within the
intent's gate budget. Findings go to `reviews/architecture_review.json` in the build
session BEFORE the gate opens; fix blocking findings and re-review. This review is
deliberately a binary inspection (each check passes or produces a severity-classed
finding), not a scored /100 rubric: forcing points onto binary completeness checks is
fake precision — the same Fagan (1976) / IEEE 1028 reasoning POSED's Stage 7 Part A
records. Scored rubrics are for quality judgments (drafted skill text); inspections are
for conformance.

## Gate

| Field | Value |
|---|---|
| gate_id | `architecture_gate` |
| decision | Two passes, one decision each: (1) stage map + gates (the shape), (2) dependency model + contested-choice threading (the wiring) |
| artifact | draft `architecture.md`; sections keyed by name |
| reviewer | fresh-context design review above (`reviews/architecture_review.json`) |
| decision_file | `architecture_gate_decision.json` — `{decision, pass, sections:[{name, disposition, comment}], guidance}` |
| owns | `page-architecture` reworks the revised sections |
| invalidates | on approval of changes: the new plugin's BUILD_PLAN ordering and any scaffold/draft output already generated (stale per umbrella rules) |
| consent | n/a — not token-expensive |

Approval of a change to `intent.md` or the grounding map after this point marks
`architecture.md` stale — say so and re-gate the affected sections. Then continue to
`page-scaffold`.
